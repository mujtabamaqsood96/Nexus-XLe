# Walkthrough: All Changes Made to the Nexus-XLe Pipeline

This document tracks all engineering changes, optimizations, and features implemented in the Nexus-XLe VLA pipeline.

---

## 1. Verification Loop (Closed-Loop Reasoning)

Transformed the pipeline from an open-loop plan generator into a closed-loop reasoning engine.

### What Changed
1. **Robust Target Extraction**: Regex noun phrase extraction and fallback rules to extract bimanual targets from the plan (e.g. `red coffee mug`, `wooden block`).
2. **Multi-Source Evidence Accumulation**: Prevents false suspicions by accumulating scene evidence from:
   - Florence-2 Object Detection labels
   - Florence-2 detailed scene captions
   - SmolVLM2 detailed scene descriptions
3. **Florence-2 Phrase Grounding Verification**: For every target noun, queries Florence-2's phrase grounding API to locate coordinates in the camera feed.
4. **Dynamic Box Suppression & Clean Visualization**:
   - **Verified Targets** (e.g. `✅ wooden block`) drawn with green bounding boxes. Overlapping standard detection boxes are suppressed.
   - **Suspicious/Hallucinated Targets** (e.g. `screwdriver`) flagged as `⚠️ SUSPICIOUS` — **no bounding box drawn** to prevent mislabeling.
   - **Failed Targets** flagged as `❌ FAILED` — step marked UNSAFE.

### Safety Verification Tiers

| Tier | Visual | Meaning |
|---|---|---|
| ✅ VERIFIED | Green bounding box | Object confirmed by scene evidence + grounding |
| ⚠️ SUSPICIOUS | No box, text warning | Grounding found a box but no text evidence mentions this object |
| ❌ FAILED | No box, text error | Not found anywhere — step is UNSAFE |

## 2. Advanced FYP Features

### 🎯 Bimanual Grasp Point Overlay (L/R Overlay)
* For verified targets, the UI calculates bimanual grasp positions relative to grounding coordinates.
* **Left Arm**: Cyan crosshair `(+) L GRASP` on left edge of bounding box.
* **Right Arm**: Pink crosshair `(+) R GRASP` on right edge of bounding box.

### 🧩 Pixel-Level Segmentation (All Objects)
* When Object Detection detects objects, the pipeline fires off parallel pixel-level Referring Expression Segmentation queries to Florence-2.
* Overlays semi-transparent, distinct colored masks on each object.

## 3. Spatial Chain-of-Thought Prompting Framework

Implemented Dr. Ho Tatt Wei's request for spatial reasoning before planning.

### The VLM Output Format
The planner now outputs a structured `SCENE → ANALYSIS → PLAN` format:
```
SCENE: <objects and their positions>
ANALYSIS: <spatial reasoning about translation/rotation axes>
PLAN:
Step 1 | LEFT ARM: <action> [TARGET: <object>]
Step 1 | RIGHT ARM: <action> [TARGET: <object>]
Step 2 | BOTH ARMS: <action> [TARGET: <object>]
```

### Key Optimizations Made

#### 3a. Repetition Penalty Conflict (Root Cause of Hallucinations)
* **Problem**: `repetition_penalty = 1.3` was applied to prevent infinite generation loops. However, bimanual planning requires repeating tokens like `LEFT ARM`, `RIGHT ARM`, `Step 1`, and target object names. The high penalty forced the model to:
  - Skip `SCENE` and `ANALYSIS` blocks entirely
  - Hallucinate random objects (`coffee container`, `paper towel roll`) in TARGET tags to avoid repeating real object names
* **Fix**: Reduced `repetition_penalty` from `1.3` to `1.05`.

#### 3b. Scene-Grounded Planning Context
* **Problem**: The planner had no physical context about the scene layout when generating plans.
* **Fix**: Updated `plan_bimanual_action()` to accept a `scene_desc` parameter. The scene description (already generated by `describe_scene()`) is now injected into the planning prompt as a grounding anchor.

#### 3c. Prompt Length Reduction
* **Problem**: The 2.2B model was overwhelmed by a very long system prompt (~50 lines). It would forget formatting instructions or copy examples verbatim (Stochastic Parrot problem).
* **Fix**: Shortened `_BIMANUAL_CONTEXT` to ~20 lines with a single focused example and concise rules.

#### 3d. Few-Shot Example Object Collision
* **Problem**: The few-shot example in the prompt used `red mug` and `wooden block` — the same objects in the test scene. The model copied the example verbatim instead of generating a new plan.
* **Fix**: Changed example objects to `green cylinder` (objects that never appear in test scenes).

#### 3e. Bimanual Post-Processing Safety Net
* **Problem**: SmolVLM2-2.2B sometimes generates single-arm plans even when the task explicitly says "both arms", because it sees the object on one side and assigns only the nearest arm.
* **Fix**: Added `_postprocess_bimanual_plan()` function in `smolvlm_planner.py` that:
  1. Detects if the task requires bimanual coordination (regex matching "both arms", "both the arms", "bimanual", etc.)
  2. Checks if the generated plan already uses both arms
  3. If not, extracts the target object from the task text and reconstructs a proper bimanual plan with LEFT ARM + RIGHT ARM grasp steps and a BOTH ARMS lift step
  4. Preserves the model's SCENE and ANALYSIS blocks

### Input Logging
* Added input logging block in `app.py` that prints all pipeline inputs at the start of each run:
  - Text fields show `(empty)` when blank
  - Toggles show `ON` / `OFF`
